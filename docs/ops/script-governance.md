# 脚本治理规则

## 目标

`scripts/` 只承载当前仍在运行的系统能力，不承载历史迁移脚本、个人工具或一次性实验脚本。

## 当前有效分层

| 目录 | 职责 |
| --- | --- |
| `scripts/actions/` | 对外可执行的主线动作入口 |
| `scripts/lib/` | 状态和数据库内核，不直接给人手跑 |
| `scripts/services/` | HTTP、通知、桥接等外部服务 |
| `scripts/core/` | 自动驾驶与系统核心运行器 |
| `scripts/maintenance/` | 健康检查、守护、复盘 |
| `scripts/ingest/` | 索引、检索、向量入库 |
| `scripts/tools/` | 少量仍在主链上使用的运维工具 |
| `scripts/wechat/` | 当前仍保留在仓库内的发布集成 |

## 治理原则

### 1. 只保留主链脚本
只要脚本不再服务当前系统主链，就不应继续留在仓库主目录中。

### 2. 禁止重复实现
同一能力只能保留一套主实现。重复检查器、重复发布器、重复桥接器必须收敛。

### 3. package.json 只暴露主线入口
对用户公开的命令必须是稳定入口，不能把历史脚本和临时脚本继续暴露出去。

### 4. 明确副作用
主线脚本必须能回答三件事：
- 读什么
- 写什么
- 是否会对外发送、发布或修改状态

### 5. 个人工具不属于 CortexOS 主仓
账号切换、个人包装器、一次性生成器这类脚本，不应混在主仓的长期维护面内。

## 新增脚本门槛

新增脚本前，必须先判断：
1. 这是不是已有能力的变体
2. 这是不是应该并入现有脚本而不是新增文件
3. 这是不是个人工具，应该放到仓库外
4. 这是不是一次性迁移任务，做完就该删除

## 删除规则

出现以下任一情况，应优先删除或迁出：
- 只服务历史目录结构
- 只用于一次性迁移
- 与现有主线能力重复
- 只对个人工作流有意义，和 CortexOS 产品边界无关
