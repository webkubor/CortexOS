# AI前端开发规范

## 定位
这是一份给 AI 直接继承的前端开发默认规范，用于约束 AI 在前端项目中的判断、实现和交付方式。

优先级规则：
1. 如果项目已有成熟技术栈、目录结构、设计系统或工程约束，优先服从项目现状，不强推新栈。
2. 如果项目缺少明确约束，按本规范的默认基线执行。
3. 工作区或业务专项规范可以在本规范之上追加，但不能和本规范的核心原则冲突。

## 默认技术栈基线
当项目没有明确约束时，默认采用：
- 前端框架：`Vue 3`
- 构建工具：`Vite`
- 语言：`TypeScript`
- 路由：`vue-router`
- 组件形态：`Vue SFC`
- 样式：`SCSS`
- 国际化：`vue-i18n`
- 时间处理：`dayjs`
- 网络层：优先 `ky` 或统一请求封装
- 运行环境：`Node.js LTS`，推荐 `22.x`

如果项目已存在 React、Next.js、Nuxt、Svelte 等成熟栈，AI 的职责是延续既有体系，不做无必要迁移。

## 核心原则
- 先读代码，再做决定；禁止脱离仓库现状凭空设计架构。
- 先复用，再新增；优先复用现有组件、样式变量、工具函数和交互模式。
- 先类型，再实现；接口、数据结构、状态边界必须先清楚。
- 先主流程，再边界；优先保证首屏、核心路径、错误路径可用。
- 先一致性，再炫技；不为展示能力引入额外框架、复杂抽象或大改目录。

## 语言与文件边界
- 业务代码默认使用 `TypeScript`
- 页面和组件优先使用 `.vue` 或项目既有组件格式
- 样式优先使用项目既有方案；若无约束，默认 `.scss`
- 类型定义优先放在 `src/types`、`src/interface` 或项目现有类型目录
- 纯工具函数放在 `src/utils`
- 可复用组合式逻辑放在 `src/composables`
- 常量、枚举、配置项放在 `src/constants`
- 国际化资源放在 `src/locales`

## 组件与页面规范
- 单个组件只负责单一职责，避免页面级巨型组件持续膨胀。
- 展示组件与业务流程逻辑尽量分离。
- 能通过 `props`、事件、插槽解决的问题，不要过早上全局状态。
- 公共组件命名必须可读，避免 `Test`、`New`、`Temp` 这类临时命名长期留存。
- 页面结构优先保证移动端可用，再处理桌面扩展。

## 代码膨胀与硬阈值约束 (200/500/8 准则)
为了防止 AI 在迭代过程中无限制堆叠代码，强制执行以下定量约束：

### 1. 硬阈值限额
- **单文件总行数 (SFC)**：禁止超过 **500 行**（含 template/script/style）。若超过，必须拆分为子组件。
- **逻辑块行数 (Script Setup)**：禁止超过 **200 行**。若逻辑复杂，必须抽离为 `composables/useXXX.ts`。
- **样式行数 (Scoped CSS)**：禁止超过 **100 行**。若样式过多，必须抽离为同名 `.scss` 文件。
- **状态复杂度**：单个组件内的 `ref`/`reactive` 状态禁止超过 **8 个**。超过此阈值说明逻辑应抽离或 Store 化。
- **计算与监听**：单个组件内 `computed` 不超过 **5 个**，`watch` 不超过 **3 个**。

### 2. 复杂组件目录化模式
对于页面级组件或预判逻辑复杂的组件，禁止直接创建 `ComponentName.vue`，必须采用目录化结构：
```text
ComponentName/
├── index.vue        (只留 Template 和基础 Entry，< 100 行)
├── useLogic.ts      (承载业务逻辑、状态、Composables 组合)
├── styles.scss      (独立样式文件)
├── types.ts         (当前组件特有的类型定义)
└── constants.ts     (大型配置项、静态文案、映射表)
```

### 3. 强制抽离触发器
- **数据预处理**：禁止在组件内定义大型 Object/Array。所有超过 10 行的静态配置必须进入 `constants.ts`。
- **逻辑复用**：任何被 2 个以上组件引用的非 UI 逻辑，必须进入 `src/composables`。
- **格式化工具**：所有涉及数据转换、时间格式化、金额计算的逻辑，禁止写在组件内，必须进入 `src/utils`。

## 外科手术式重构协议 (Surgical Refactoring Protocol)
为了解决“拆分即崩溃”的风险，AI 在面对超过阈值的巨型组件时，必须遵循以下步骤：

### 1. 逻辑先行：剥离 Composition API (Brain-First)
- **禁止**直接将 HTML 模板剪切到新组件。
- **优先**创建 `useComponentLogic.ts`，将原组件 `<script setup>` 中的状态（ref/reactive）、计算属性（computed）和业务函数搬移过去。
- **验证点**：确保原组件通过 `import { useComponentLogic } from './useComponentLogic'` 依然能正常运行。

### 2. 契约驱动：定义接口 (Contract-First)
- 在拆分 UI 子组件前，必须先在 `types.ts` 中定义该子组件的 `Props` 和 `Emits`。
- **强制要求**：AI 必须先输出子组件的接口定义，并确认父组件能提供这些数据。

### 3. 中枢调度：index.vue 降级 (The Bridge Pattern)
- 重构后的 `index.vue` 应当成为一个单纯的“调度中心”。
- **职责限制**：`index.vue` 只负责引用 `useLogic` 获取状态，并将其分发给各子组件。
- **行数要求**：重构后的 `index.vue` 模板部分应当由一系列清晰的子组件标签组成，禁止保留复杂的原生 HTML 嵌套。

### 4. “防崩”分步验证清单
- **状态同步检查**：拆分后，确保子组件对 `props` 的修改是通过 `emit` 告知父组件，而不是直接修改（保持单向数据流）。
- **CSS 隔离检查**：检查是否有样式依赖于深层嵌套的选择器（如 `.parent .child`），拆分后应改为独立的 class 或 CSS Modules。
- **生命周期对齐**：检查原组件的 `onMounted` 等生命周期是否需要拆分到子组件或保留在 `useLogic` 中。
- **局部验证**：每拆出一个子组件，必须立即进行一次局部编译/运行验证，严禁“一拆到底”后再调试。

### 5. 灾难恢复协议
- AI 在开始重构前，必须声明：`“我将开始重构 [组件名]，第一步将提取逻辑到 useLogic.ts，请确认是否保留原文件备份。”`
- 如果拆分后出现无法解决的类型错误或运行异常，AI 必须具备**回滚能力**，即能够根据备份或 Git 历史恢复到拆分前的状态。

## 注释规范
- 默认要求为每个具名函数补函数头注释；至少覆盖导出函数、组合式函数、请求封装、业务处理函数和关键工具函数。
- 函数头注释至少写清：用途、参数、返回值、关键副作用、边界条件。
- Vue 组件、`composables`、请求模块、复杂常量模块应有文件头注释，说明职责和使用边界。
- 临时匿名回调如果生命周期极短且语义清晰，可以不单独写函数头注释；其余情况不应省略。
- 禁止写无信息量注释，例如“给变量赋值”“调用接口获取数据”这类重复代码表面的描述。
- 模板入口：[[函数头注释模板]]

## 状态与数据流规范
- 优先使用组件局部状态和组合式函数管理状态。
- 只有跨页面、跨模块共享的数据，才提升到全局层。
- 请求、转换、展示三个层次尽量分离，避免在组件模板内堆积业务处理。
- 禁止在多个页面复制同一段请求逻辑或返回值处理逻辑。

## 网络层规范
- 页面和组件中不直接散落原始请求实现细节。
- 请求统一通过 `src/utils/request` 或等价封装层收口。
- 错误处理、重试、鉴权、跳转、副作用必须收敛在统一层处理。
- 新增接口时优先补齐输入、输出和错误结构的类型定义。
- 如果项目仍在使用历史请求库，AI 迁移时必须保证行为兼容，再逐步收敛。

## 样式与设计规范
- 先继承现有设计系统；如果没有，再建立最小变量体系。
- 颜色、字号、间距、圆角、阴影优先变量化，不写散落魔法值。
- 避免复制整页样式；公共视觉模式应抽成变量、混入或共享组件。
- 动效只用于强化反馈，不用于掩盖结构问题。
- 不默认引入“AI 味”很重的通用审美，优先做清晰、克制、可维护的界面。

## 国际化规范
- 用户可见文案不长期硬编码在组件中。
- 多语言项目新增文案时，至少同步默认语言和当前目标语言。
- 国家或品牌差异优先通过 `locales`、配置或映射表管理，而不是模板里大量条件分支。

## 可访问性与兼容性
- 表单控件必须具备清晰标签、状态和错误提示。
- 交互元素必须可聚焦、可点击、可反馈。
- 移动端点击热区、滚动行为、键盘弹起影响必须纳入考虑。
- 涉及旧浏览器或低性能设备时，优先选择低风险实现。

## AI 执行流程
1. 先识别技术栈、目录结构、现有模式和复用点。
2. 判断改动属于新增功能、重构、修复还是对齐。
3. 先修改类型、常量、接口和数据边界，再改页面和组件。
4. 最后补样式、国际化、埋点和文档。
5. 交付前至少做一次自检，并明确说明是否验证成功。

## 构建与环境文件规范
- 多环境前端项目默认采用 `Vite mode` 方案管理环境差异。
- 默认环境文件命名：`.env`、`.env.test`、`.env.production`。
- 默认构建脚本命名：`build:test` 对应 `vite build --mode test`，`build:prod` 对应 `vite build --mode production`。
- 如果项目已有更严格脚本，例如 `vue-tsc -b && vite build --mode test`，应在保留 `build:test` / `build:prod` 语义的前提下增强，不要改乱命名口径。
- 环境差异必须通过 env 文件、构建 mode 或统一配置收口，禁止把 test / production 差异硬编码在业务代码里。
- `.env` 放公共默认项，`.env.test` 放测试环境差异，`.env.production` 放正式环境差异。
- 任何新增环境变量都必须同步说明用途、默认值、使用场景，并检查 test / production 是否都已补齐。

## 版本号与版本记录规范
- 任何可部署、可回滚、可感知的前端变更，都必须纳入版本号管理。
- 版本号以项目主版本文件为准；前端项目默认以 `package.json` 的 `version` 作为源码侧基线。
- 版本递增遵循最小可理解原则：修复用补丁版本，兼容性新增用次版本，破坏性变更用主版本。
- 每次发布必须保留版本记录，不允许只覆盖线上产物而不留版本痕迹。
- 版本记录至少包含：版本号、日期、变更摘要、影响模块、风险说明、回滚入口。
- 推荐维护 `CHANGELOG.md`、`docs/releases/` 或等价发布记录；团队一旦选定一种方式，后续必须连续使用。
- 发布脚本、产物目录、回滚脚本和通知文案应尽量引用同一版本号，避免“代码版本、构建版本、发布版本”三套口径分裂。
- 模板入口：[[版本记录模板]]

## 质量门禁
提交前至少满足以下条件中的适用项：
- 类型检查通过
- 构建通过
- 关键主流程可手动验证
- 没有新增硬编码文案、重复请求逻辑或明显样式分叉
- 变更说明能说清“改了什么、为什么改、风险在哪里”
- 若本次变更进入发布链路，已同步版本号与版本记录
- 审查输出优先按 [[代码审查输出模板]] 组织

## 明确禁止
- 未阅读现有代码就大面积重写
- 未确认复用点就新建重复组件或重复工具函数
- 在页面组件中堆积请求、数据转换、样式分支和文案常量
- 为了“现代化”强推与项目现状冲突的框架迁移
- 未验证主流程就宣称完成

## 与专项规范的关系
工作区规范、项目规范、设计规范都可以在本规范之上补充约束。
例如：[[收银台工作区开发规范]] 负责定义该工作区的项目分层和业务特化边界；它不替代本规范。
