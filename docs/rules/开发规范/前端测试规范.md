# 前端测试规范

## 1. 定位与原则

### 1.1. 定位

本规范定义的是**代码层面的测试**，由**开发人员**编写，用于保障代码质量、支持重构、并加速开发迭代。它与测试部门负责的端到端（E2E）黑盒测试互为补充，共同构成项目的质量保障体系。

**核心目的**：在代码提交前，从源头发现并修复逻辑错误，降低质量成本。

### 1.2. 核心原则

- **TDD 优先（测试驱动开发）**：先写测试，再写实现。这不仅仅是测试方法，更是优秀的设计方法。
- **谁开发，谁测试**：编写代码的开发者是编写单元/集成测试的第一责任人。
- **测试是交付物的一部分**：提交功能时，测试代码与业务代码必须一同提交。
- **只测“做什么”，不测“怎么做”**：测试应关注函数的输入输出和组件的行为，而不是内部实现细节。

---

## 2. 测试类型与分层

我们采用经典的测试金字塔模型：

| 测试类型 | 职责 | 示例 | 占比 |
|---|---|---|---|
| **单元测试 (Unit)** | 验证最小代码单元（函数、组件）的逻辑 | `calculateFee()` 函数的返回值；按钮在 disabled 状态下不可点击 | **70%** |
| **集成测试 (Integration)** | 验证多个单元协同工作的正确性 | 支付表单提交后，是否正确调用了 API 模块 | **20%** |
| **E2E 测试 (End-to-End)** | （由测试部门负责）模拟真实用户操作 | - | 10% |

**本规范主要约束单元测试和集成测试。**

## 4. 规范细则

### 4.1. 文件命名与位置

- 测试文件必须与被测试文件同级，或放在 `__tests__` 目录下。
- 命名：`<被测文件名>.spec.ts` 或 `<被测文件名>.test.ts`。

```
// 推荐同级存放，便于查找
components/
├── PaymentButton.vue
└── PaymentButton.spec.ts

utils/
├── format.ts
└── format.spec.ts
```

### 4.2. 强制 TDD（测试驱动开发）流程

对于所有新功能和 Bug 修复，必须遵循“红-绿-重构”的流程：

1.  **【红】写一个失败的测试**：
    - 针对即将开发的功能或要修复的 Bug，写一个测试用例。
    - 运行测试，确认它因为功能未实现或 Bug 存在而**失败**。这是为了确保测试本身是有效的。

2.  **【绿】写最少的代码让测试通过**：
    - 编写刚好能让测试用例通过的业务代码。
    - 此时不追求代码优雅或性能，目标是快速让测试变绿。

3.  **【重构】优化代码**：
    - 在测试的保护下，对刚刚编写的代码进行重构，提升可读性、性能和结构。
    - 每次重构后都运行测试，确保没有破坏任何功能。

### 4.3. 质量门禁：最低测试覆盖率

- **新代码提交**：合入主干前，必须保证本次改动所涉及的文件**行覆盖率（Lines）不低于 80%**。
- **存量代码**：鼓励在修改旧代码时，顺手补充单元测试，逐步提升整体覆盖率。
- CI/CD 流水线会集成覆盖率检查，不达标的 PR 将被**自动阻断**。

### 4.4. `describe`, `it`, `test` 描述规范

- `describe(被测单元, () => { ... })`：描述被测试的模块或组件。
- `it(期望的行为, () => { ... })` 或 `test(期望的行为, () => { ... })`: 用自然语言清晰描述这个测试用例验证的**行为**和**期望结果**。

```typescript
// GOOD
describe('formatAmount', () => {
  it('should format number to string with two decimal places', () => {
    expect(formatAmount(123)).toBe('123.00');
  });

  it('should handle string input', () => {
    expect(formatAmount('456.7')).toBe('456.70');
  });
});

// BAD - 描述不清
describe('test format', () => {
  it('test', () => {
    // ...
  });
});
```

### 4.5. Mock 规范

- **API 请求**：必须 Mock。单元测试不应发出真实的网络请求。使用 `vi.mock` 模拟整个 API 模块。
- **第三方库**：如 `vue-router`、`pinia`，通常使用官方提供的测试工具或 `vi.mock` 来模拟。
- **日期与时间**：对于依赖当前时间的函数，应使用 `vi.useFakeTimers()` 来固定时间，确保测试的可重复性。

### 4.6. 禁止事项

- **禁止为了覆盖率而写无意义的测试**：测试必须有明确的断言（`expect`）。
- **禁止测试依赖外部环境**：单元测试必须是独立的，能在任何没有网络、没有数据库的环境下运行。
- **禁止一个 `it` 中包含多个不相关的断言**：一个测试用例只验证一个行为。
- **禁止在业务代码中留存测试用的假数据或逻辑**。

## 5. 项目集成

### 5.1. `package.json` 脚本

```json
{
  "scripts": {
    "test": "vitest",
    "test:unit": "vitest --mode test",
    "coverage": "vitest run --coverage"
  }
}
```

### 5.2. `vite.config.ts` (或 `vitest.config.ts`)

```typescript
/// <reference types="vitest" />
import { defineConfig } from 'vite';

export default defineConfig({
  test: {
    globals: true, // 允许全局使用 describe, it, expect 等
    environment: 'jsdom', // 模拟 DOM 环境，用于组件测试
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json', 'html'],
      lines: 80, // 最低行覆盖率
      branches: 80, // 最低分支覆盖率
      functions: 80, // 最低函数覆盖率
      statements: 80, // 最低语句覆盖率
    },
  },
});
```
