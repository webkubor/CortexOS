# 代码审查规范（Code Review）

> 目标：让代码审查结果稳定、可执行、可追踪，避免“只提感受，不提风险”。

## 1. 审查优先级

- **先风险后风格**：优先检查功能正确性、回归风险、数据安全、边界条件。
- **风格问题次级**：在不影响正确性的前提下，再处理命名、排版、注释一致性。

## 2. 必查清单（每次审查都要过）

1. 功能是否按需求工作（含主流程与异常分支）。
2. 是否引入回归风险（兼容旧数据、旧接口、旧交互）。
3. 是否有安全风险（秘钥泄露、越权、注入、敏感信息落盘）。
4. 是否有可维护性风险（过长函数、重复逻辑、隐式耦合）。
5. 是否补充了必要测试或验证步骤。

## 3. 结论分级

- **P0（阻断）**：会导致错误结果、数据损坏、安全事故，必须修复后再合并。
- **P1（高）**：高概率线上问题或核心体验损坏，建议本次修复。
- **P2（中）**：中等风险或可维护性明显下降，可排期但需建跟踪项。
- **P3（低）**：优化项或风格建议，不阻断合并。

## 4. 评论格式（统一输出）

- **问题**：一句话说明风险点。
- **影响**：会影响什么场景/用户/数据。
- **建议**：最小修复方案。
- **定位**：文件路径 + 行号。

示例：

```markdown
[P1] 空值分支未处理
- 影响：当接口返回空数组时，页面渲染报错。
- 建议：在 map 前增加空值守卫并补一个单测。
- 定位：src/modules/order/list.ts:42
```
