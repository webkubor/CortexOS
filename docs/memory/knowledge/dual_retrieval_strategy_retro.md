# 🧠 AI Common 大脑检索架构复盘：双轨制知识获取协议 (Dual Retrieval Strategy)

> **记录时间**: 2026-02-28
> **标签**: `架构设计`, `RAG`, `ChromaDB`, `AI-Agent`, `知识检索`

## 1. 架构提出的背景 (背景冲突)

在日常的 AI (如本 Agent) 操作中，经常面临“知识搜寻”的需求。如果我们一味追求高大上的 RAG (检索增强生成) 或者向量检索 (基于 Ollama + ChromaDB)，会导致严重的问题：

- **算力开销过大**：每一次向本地模型请求 Embeddings 切片相似度，都在无端压榨 Mac 本机的性能。
- **响应高延迟**：启动 Python 进程、加载引擎并进行密集运算，一次查询往往增加 1~3 秒阻塞，使得本来可以瞬间完成的代码查阅变得卡顿。
- **上下文噪音（幻觉风险）**：向量检索本质是“语义联想”。当指令非常明确时（如“修改脚本第 10 行”），盲目使用 RAG 搜索会返回大量包含同义词却毫无逻辑联系的历史数据，挤占 Token 上下文，甚至误导 Agent。

## 2. 核心解法：双层降级制体系

基于此，本系统正式确立并向全系 Agent 颁布了**“双轨制知识获取协议”**（已写入 `README.md`）。

### 轨迹一：物理精准扫盘 (Primary Fast Path)

- **适用场景**：明确的代码任务、精准的文档路径探查、已知参数名寻找。
- **依赖工具**：底层原生工具 `grep_search`, `view_file`, `find_by_name` 等。
- **机制与收益**：0延迟、0额外能耗、100%字符级精准查杀。面对工程侧的“精准执行命令”，这是**唯一的首选项**。

### 轨迹二：ChromaDB 语义检索 (Fallback / Fuzzy Query Path)

- **适用场景**：只记得“意图”但不确切记得“关键词”的历史记忆（如“之前那个图床怎么弄的来着”、“某个复盘记录”等）。
- **依赖工具**：底层猎犬脚本 `uv run python3 scripts/ingest/query_brain.py "自然语言模糊词"`。
- **机制与收益**：调用 `nomic-embed-text` 模型，以高算力和一定时间成本为代价，提供真正拥有理解能力的知识拼图关联（RAG）。
- **红线约束**：它必须被作为**“兜底方案 (Fallback)”**。任何 Agent **严禁**在日常查代码、翻具体文件名时去滥用该指令。

## 3. 运维收益

随着这套防线与调用协议的沉淀，整个 `AI_Common` 外部大脑体系拥有了兼顾赛博机械速度与 AI 直觉思维的**最强工作流流派**。既保障了日常代码的毫秒级推进，又在“脑子断片”时拥有了坚实的记忆保底。
