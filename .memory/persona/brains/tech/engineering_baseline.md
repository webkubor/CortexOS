# 工程基线

> 这是 CortexOS 的统一工程规则入口。编码规范、Review 规范、提交规范、环境基线统一收敛到本文件。

## 1. 适用范围

适用于所有代码、脚本、配置、SQL、文档与运维修改。

## 2. 运行时基线

- 操作系统基线：macOS
- Python 包管理与执行：统一使用 `uv`
- Node 包管理：统一使用 `pnpm`
- Node 版本基线：以仓库 `.nvmrc` 为准
- GitHub 自动化：优先使用 `gh` CLI

## 3. 可用代码标准

- 代码必须能在当前项目真实环境中运行、验证或被最小 smoke test 覆盖
- 关键路径必须有明确输入、输出、失败行为
- 不允许“先占位、后补实现”的伪完成状态进入主线
- 改动后必须给出验证方式

## 4. 明确禁止

- 垃圾代码：复制粘贴后只改变量名、死分支不删、无用参数不清
- 胶水代码：临时串命令、串脚本、串文件格式，没有边界、没有错误处理、没有回收计划
- 屎山代码：同一事实多份状态源、旧逻辑不删只绕开、修一处要碰多处无关模块
- 伪抽象：没有明确第二复用点就提前抽 helper / manager / 基类
- 静默失败：吞异常、返回假成功、日志没有关键上下文
- 配置污染：把路径、环境差异、实例信息硬编码进业务逻辑

## 5. 结构与依赖规则

- 一个模块只解决一个稳定问题
- 真理来源必须唯一；展示层文件默认都是视图，不是状态源
- 配置读取必须集中管理，不能在多个模块里各自猜路径和默认值
- 关键链路优先减少跨语言、跨进程、跨存储跳转
- 高风险依赖（原生模块、系统命令、跨 runtime 调用）只有在收益明显大于维护成本时才允许进入关键路径

## 6. 函数注释标准

- 所有导出函数、公共方法、脚本入口函数、跨模块调用函数、异步流程函数，必须写标准函数注释
- 复杂私有函数只要包含业务判断、状态变更、外部调用、降级逻辑，也必须写标准函数注释
- 只有极短、无分支、语义一眼可懂的局部回调，才允许省略函数注释
- 函数注释不是翻译函数名，而是解释“为什么存在、接什么、返回什么、可能怎么失败”
- 注释格式遵循语言习惯：TypeScript / JavaScript 用 JSDoc 或 TSDoc，Python 用 docstring，其他语言用该生态主流 doc comment 形式
- 注释语言统一用中文；类型、参数名、异常名保留代码原文

函数注释最少必须说明：

- 功能：这个函数解决什么问题
- 参数：关键输入含义、约束或默认值
- 返回：返回值含义，是否可能为空
- 副作用：是否写文件、改状态、发请求、写数据库、触发外部系统
- 失败行为：抛错、返回空、降级路径或重试边界

禁止写法：

- 只写“处理数据”“执行逻辑”这种空话
- 参数很多但不解释关键约束
- 有副作用却不写副作用
- 明明会失败却把异常路径藏起来
- 复制模板注释但和实际实现不一致

推荐模板：

```ts
/**
 * 作用：同步当前任务状态到舰队看板，并在必要时触发降级提示。
 * @param taskId 任务唯一标识，不能为空。
 * @param silent 是否静默模式；为 true 时只写日志不弹提示。
 * @returns 返回同步是否成功，以及是否走了降级路径。
 */
```

```py
def sync_dashboard(task_id: str, silent: bool = False) -> dict:
    """
    作用：同步当前任务状态到舰队看板，并在必要时触发降级提示。
    参数：
      task_id: 任务唯一标识，不能为空。
      silent: 是否静默模式；为 True 时只写日志不弹提示。
    返回：
      包含同步结果与是否走降级路径的字典。
    """
```

## 7. 错误处理与验证

- 所有外部调用都必须考虑失败：文件、权限、网络、依赖、版本、空结果
- 错误信息至少包含动作、对象、失败原因
- 可以降级时允许 fallback，但必须显式说明当前不是主路径成功
- 重试必须有边界
- 没有测试框架也必须给出最小验证路径
- 修 bug 至少验证“原问题消失”和“核心旧行为未回退”

## 8. 试错与复盘升级协议

- 允许在方案层做小步试错，但每次试错都必须有边界、验证和回收动作
- 不要因为害怕出错而放弃更优解；要把试错结果沉淀成下次可复用的资产
- 一次性噪音问题只记任务日志，不升格为规则
- 重复 2 次以上的技术坑，或明显可复用的修复套路，升级到 `.memory/persona/retry_patterns.md`
- 一次就代价很高、影响关键链路、暴露系统性缺陷的问题，直接升级到 `.memory/retros/`
- 对项目方法、架构判断、业务方案有长期价值的结论，写入 `memory/knowledge/`

最小复盘事实必须包含：

- 症状
- 触发条件
- 根因
- 修复动作
- 验证命令

回注原则：

- 下次遇到相似问题时，优先读取相应 `retry_patterns` / `retros`
- 如果同类问题已被复盘过，不允许重复靠临场记忆硬猜

## 9. Code Review 基线

- 先风险后风格
- 每次审查至少检查：
  - 功能正确性
  - 回归风险
  - 安全风险
  - 可维护性风险
  - 测试或验证是否充分

风险分级：

- `P0`：错误结果、数据损坏、安全事故，必须修复
- `P1`：高概率线上问题或核心体验损坏，建议本次修复
- `P2`：中等风险或明显维护性下降，需要排期
- `P3`：低风险优化项

统一评论格式：

- 问题
- 影响
- 建议
- 定位（文件路径 + 行号）

## 10. 提交规范

提交信息格式：

`<type>(<scope>): <subject>`

常用 `type`：

- `feat`
- `fix`
- `docs`
- `refactor`
- `perf`
- `test`
- `chore`
- `revert`

要求：

- `subject` 使用中文
- 动词开头
- 不超过 50 字

## 11. 提交前自查

- 这次改动现在能跑吗
- 有没有引入第二份状态源
- 有没有把复杂度只是从 A 文件搬到 B 文件
- 有没有留下无清理计划的临时兼容层
- 出错时能否在 5 分钟内通过日志定位方向
- 是否给出明确验证步骤
- 如果这次踩坑了，是否已经判断它该进日志、retry、retros 还是 knowledge
- 导出函数、公共方法和复杂流程函数，是否已经补上标准函数注释
