# 代码标准技能

这个 skill 不是只约束某个框架，也不是只做“代码审查”。

它的核心定位是：
- 以 **TypeScript** 为基座
- 对 **Vue / React / 通用 TS 模块** 都有效
- 约束代码如何更易维护、更美观、更一致、更适合长期演进
- 把 **SSOT（Single Source of Truth）**、交互一致性、模块边界、类型契约一起纳入标准

当用户要求以下事情时使用：
- 审查代码质量、可维护性、长期演进风险
- 为项目建立或更新编码标准
- 降低复杂度、缩小文件体积、拆分大模块
- 强化 TypeScript 约束、类型边界、状态边界
- 统一 Vue / React 的交互模式、组件职责、数据流
- 解决“代码不好维护 / 看起来乱 / 容易漂移 / 多处真源”的问题

请勿将此技能用于：
- 品牌视觉或 Logo 设计
- 单纯营销文案润色
- 与维护性无关的样式争论
- 纯脚手架初始化（除非用户明确要建立代码标准）

## 核心原则

### 1. TypeScript 是第一约束层
- 优先让类型表达业务边界，而不是只给编辑器补提示
- 类型要描述真实契约，不要用 `any`、模糊对象或过宽联合掩盖结构问题
- 让 `props / params / returns / domain model / API payload` 都有明确边界
- 类型定义优先放在稳定边界，避免散落重复声明

### 2. SSOT 必须明确
- 一个状态、配置、规则、映射，原则上只能有一个真源
- 不允许多个组件、多个 hooks、多个 stores 各自维护相同事实
- UI 展示、派生值、格式化值应来自真源推导，而不是复制一份
- 如果发现“多处同步更新才能正确”，优先收口真源

### 3. 框架只是壳，交互逻辑要可维护
- Vue 和 React 都要把：
  - 展示
  - 状态
  - 副作用
  - 交互编排
  拆清楚
- 不把复杂业务判断直接塞进模板 / JSX
- 不让事件处理函数变成隐形业务引擎
- 一切交互逻辑都应能被单独理解、单独测试、单独替换

### 4. 美观不是装饰，是结构清晰
- “好看”的代码首先是可扫描、可预测、可修改
- 模块命名、文件布局、导出边界、状态流向应有秩序感
- 代码风格服务于理解，而不是制造“高级感”

### 5. 为维护而优化，而不是为了技巧
- 相比花哨抽象，更偏向显式流程
- 相比框架技巧，更偏向稳定契约
- 相比全量重写，更偏向最小可行重构

## 默认审查维度

### TypeScript 维度
- 类型是否真实表达数据结构和边界
- 是否存在 `any / as unknown as / 超宽 Record<string, any>` 这类逃逸
- 类型定义是否重复、漂移或和运行时事实不一致

### SSOT 维度
- 数据真源在哪里
- 有没有重复状态、重复映射、重复规则
- 派生值是否由真源计算，而不是手动同步

### 结构维度
- 一个模块是否只有一个主要职责
- 文件是否混合了 UI、状态、请求、副作用、业务编排
- 命名是否表达“它负责什么”，而不是“它怎么实现”

### 交互维度
- 交互状态是否集中管理
- 事件处理是否清晰可追踪
- 用户操作链是否存在多处隐式触发和难以定位的副作用

### 框架维度
- Vue：`template / composables / state / side effects` 是否边界清楚
- React：`components / hooks / derived state / effects` 是否职责清楚
- 两者都避免“组件即一切”的反模式

## 默认启发式规则

- 超过约 `500` 行的 Vue SFC 或 TSX 页面应被质疑
- 超过约 `200` 行的交互逻辑块应被质疑
- 超过约 `8-10` 个核心响应式状态，通常意味着需要分解
- 如果一个组件同时拥有：
  - 远程请求
  - 状态机
  - 表单校验
  - 复杂渲染
  - 多个副作用
  那它大概率职责过载
- 同一领域对象如果在多个文件重复定义 shape，优先建立统一类型真源

这些是审查阈值，不是机械规则。始终按上下文判断。

## Vue / React 具体要求

### Vue
- 复杂页面优先把业务逻辑提到 composables 或 domain services
- `watch / computed / ref` 数量过多时，优先检查是否真源分散
- 避免模板层出现大段业务分支

### React
- 避免把副作用、派生状态、交互编排全部塞进一个组件
- `useEffect` 不应用来偷偷同步多个状态真源
- 大量 props drilling 或上下文塞满业务状态时，应重新审视边界

### 通用 TypeScript 模块
- 纯逻辑优先保持框架无关
- domain types、formatters、mappers、validators 尽量从 UI 层抽离
- 让可复用逻辑独立于 Vue/React 生命周期

## 输出契约

使用此技能时：
- 先指出真实的维护风险，而不是泛泛而谈
- 先判断真源和边界，再讨论拆分
- 明确指出是：
  - 类型问题
  - SSOT 问题
  - 结构问题
  - 交互问题
  - 框架使用问题
- 提出能显著改善代码的最小可行重构方案
- 如果用户要求审查，优先指出缺陷、回归风险、缺失测试和未来维护成本

## 约束

- 不为了好看而做无意义重排
- 不只围绕单一框架写规则
- 不为了拆分而拆分
- 不把抽象模式当成默认正确答案
- 不允许 SSOT 漂移被“临时兼容写法”长期掩盖
