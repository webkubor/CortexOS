# 代码规范技能

此技能旨在保持代码库的可读性、可维护性和结构合理性。

当用户要求执行以下操作时使用：
- 审查代码质量
- 降低复杂度或缩小文件体积
- 重构大型组件或模块
- 强制执行命名、结构和副作用规范
- 在功能开发前后加强可维护性

请勿将此技能用于：
- 品牌或徽标设计
- README 设计润色
- 通用项目脚手架
- UI 美学（除非它们直接影响可维护性）

## 核心规则

### 1. 为维护而优化，而非为了“奇技淫巧”
- 相比于神奇的抽象，更倾向于显式的流程
- 减少隐式耦合
- 将业务逻辑与框架连接代码及显示代码分离

### 2. 当文件不再易于导航时进行拆分
- 只有在大型文件仍易于理解时才是可接受的
- 当一个文件混合了编排、状态、渲染和副作用时，按职责进行拆分
- 在跨组件重复修复之前提取共享逻辑

### 3. 使用具体的阈值进行审查
- 文件大小和函数大小
- 状态数量和响应式复杂度
- 副作用范围
- 命名清晰度
- 可测试性和契约清晰度

### 4. 增量重构
- 如果针对性的提取已足够，请不要重写一个正在运行的模块
- 先保证行为不变，再改进结构
- 在仓库支持的情况下，在每个结构化步骤后进行验证

## 默认启发式规则

- 超过约 500 行的 Vue 单文件组件（SFC）应受到质疑
- 超过约 200 行的组件逻辑块应受到质疑
- 单个组件中过多的 ref、watcher 或 computed 值通常预示着缺乏分解
- 如果一个函数需要大量的解释，那么其结构可能过于紧密

这些是审查阈值，而非自动执行的定律。请根据实际情况判断。

## 输出契约

使用此技能时：
- 首先指出真实的可维护性风险
- 指出结构性问题，而不仅仅是表象
- 提出能显著改进代码的最小可行重构方案
- 如果用户要求审查，请优先考虑缺陷、回归和缺失的测试，而非赞美

## 加载规则

- 始终阅读 `references/coding-standard.md`

## 约束

- 不要强加无法改善维护性的风格更改
- 不要仅仅为了满足某个数字而拆分文件
- 除非复用价值足以证明其合理性，否则不要用抽象模式替换清晰的局部逻辑
